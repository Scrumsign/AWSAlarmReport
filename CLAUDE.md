# AWSAlarmReport — プロジェクトガイド

AWS CloudWatch アラームを受信し、Bedrock で原因解析して Discord に通知する Lambda。

## 作業方針

- 問題・懸念点（型の不整合、設計との乖離、破壊的変更の可能性など）は即座にユーザーに伝える
- 実装は慎重に進め、確認が取れたステップから順に着手する

## テスト・操作時の確認ルール

以下に該当する場合は、**必ずユーザーに確認を取ってから実行する**。

**確認必須（外部への副作用あり）**:
- デプロイ・ビルド（docker build / push、Lambda 更新、GitHub Actions 実行 など）
- AWS リソースへの変更（IAM ポリシー追加、SES 設定変更、CloudWatch 操作 など）
- 外部サービスへのリクエスト（SES メール送信、Discord Webhook 送信、Bedrock API 呼び出し など）
- e2e テスト・疎通テスト（実環境に対して実際のリクエストを送るもの）

**確認必須（グレーゾーン・判断しかねるとき）**:
- 操作の副作用・影響範囲が不明確なとき
- 実環境 / ステージング環境 / ローカル環境の区別がつかないとき
- TEST.md の種別が e2e・手動・integration など、unit 以外を含むとき

確認なしに実行してよいのは、外部への副作用を持たない **unit テスト・static 解析・ローカルのファイル操作** のみ。

**判断に迷ったときの調査ルール**:
- 確認が不要と明確に判断できる場合は調査を省略してよい
- 確認が必要と判断した場合、またはグレーゾーンで判断しかねる場合は、**ユーザーへの確認前に必ずウェブで調査**する（操作の副作用・影響範囲・ベストプラクティスを把握してから確認を求める）
