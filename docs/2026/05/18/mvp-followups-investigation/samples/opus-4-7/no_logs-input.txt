# 監視対象 Lambda の運用前提 (不変)
- **トリガー**: 4 時間に 1 回、特定 S3 バケットへの ZIP アップロード
  (例 `inputFiles/<ship_name>-<ship_timestamp>.zip`) を契機に起動する
- **1 起動 = 1 ファイル処理**: ship_name (船名) + ship_timestamp で識別される
  単一 ZIP を入力として処理する
- **エラー判定基準**: 直近時間窓 (Alarm 発火時刻 ±N 分) 内に
  `status="success"` のログが **1 件も存在しない** こと
- **したがってエラーの根本パターンは 2 つ**:
  - (A) **起動形跡なし** = error / success どちらのログもない。
        S3 アップロード自体がなく Lambda が起動していない可能性が最有力
  - (B) **起動して失敗** = status="error" のログが存在し exception/stack_trace が取れる。
        コードバグ / 入力データ異常 / 設定欠落 / 外部依存障害のいずれか

直近時間窓内に error / success どちらのログも存在しない。
→ 最有力仮説: **S3 への ZIP アップロード自体がなく Lambda が起動していない**。
  考えられる原因:
    - 上流のアップロード処理 (船側送信 / 別 Lambda / 手動転送) の失敗・遅延
    - ネットワーク障害でアップロードが届かなかった
    - S3 イベント通知設定の破損で Lambda が起動しなかった
→ 副次仮説: Lambda が cold start で powertools logger 初期化前にクラッシュし、
  error ログも success ログも出力されないケース (頻度は低い)。
→ ログ配信遅延 (CloudWatch Logs subscription / throttle) で実際は処理済みだが
  ログだけ届いていないケースも可能性として残す (確認手段: CloudTrail / Lambda 直接 Metric)。
suggested_actions は S3 オブジェクト着信確認 / CloudTrail 突合 /
