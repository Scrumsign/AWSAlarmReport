# [Enhancement] 通知メッセージを非技術者にも理解できる内容・形式にする

GitHub Issue: #4

## 背景・問題

現在の通知メッセージ（Discord / メール）は、AWS の技術用語（Lambda、Alarm、AssumeRole、CloudWatch Logs Insights など）や内部設定ファイル名がそのまま表示されており、技術者以外の受信者には何が起きているのか伝わらない。

また、メッセージが伝えているのは「技術的に何が発生したか」であり、「業務上どういう状態か・何に影響があるか」が書かれていない。例えば「AssumeRole 失敗」ではなく「ファイル処理の結果が確認できない状態です」のように、受信者の関心に合った表現が必要。

一方で技術者もこの通知を読むため、技術的な詳細（アラーム名、エラー種別、原因の詳細など）は引き続き含める必要がある。

## ユーザーストーリー

通知の受信者（技術者・非技術者を問わない）として、
アラーム通知を受け取ったときに、業務上何が起きていて何に影響があるかをまず理解し、必要に応じて技術的な詳細も確認したい。
なぜなら、対応の要否や緊急度を受信者自身が素早く判断できるようにしたいから。

## 対象範囲

- Discord embed（通常通知 `_to_embed` / fallback 通知 `_post_minimal_embed`）
- Email 通知（HTML / プレーンテキスト）
- Bedrock プロンプトの出力制約（LLM が生成する root_cause / suggested_actions の表現）

## 受け入れ条件

- [ ] 通知の冒頭で「業務上何が起きているか」が平易な日本語で示されること（例: 「ファイル処理が開始されていない可能性があります」）
- [ ] 技術的な詳細（アラーム名、エラー種別、AWS 固有の情報など）は補足として通知内に含まれること
- [ ] フィールド名・ラベルから AWS 固有の用語が排除されていること（Lambda、Alarm、AssumeRole 等がラベルに直接出ない）
- [ ] fallback 通知の文言から内部設定ファイル名（`config/alarm_log_groups.yml` 等）や AWS API 名が排除されていること
- [ ] severity / confidence が日本語で表現されていること（例: HIGH → 重大、confidence: high → 確信度: 高）
- [ ] Bedrock の出力（原因仮説・推奨アクション）にも業務的な説明が含まれるようプロンプトが調整されていること
- [ ] 非技術者が通知を読んだだけで「対応が必要か」「誰に相談すべきか」を判断できること
