## TASK-001: クライアントアカウントに SNS Topic を作成し Topic Policy を設定する

- **対応要件**: REQ-002

クライアントアカウント (920373030024) で CloudWatch Alarm 用 SNS Topic を作成し、
Topic Policy で以下を許可する:

- `cloudwatch.amazonaws.com` の `SNS:Publish` (同一アカウント内 Alarm のみ)
- `arn:aws:iam::088898720463:root` の `SNS:Subscribe` / `SNS:Receive`

作成コマンド:

```
aws sns create-topic --name hdw-ml-alarm-topic \
  --profile hanshin-t.kimura --region ap-northeast-1
```

Topic Policy (`set-topic-attributes` で適用):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudWatchPublish",
      "Effect": "Allow",
      "Principal": { "Service": "cloudwatch.amazonaws.com" },
      "Action": "SNS:Publish",
      "Resource": "arn:aws:sns:ap-northeast-1:920373030024:hdw-ml-alarm-topic",
      "Condition": {
        "ArnLike": {
          "aws:SourceArn": "arn:aws:cloudwatch:ap-northeast-1:920373030024:alarm:*"
        },
        "StringEquals": {
          "aws:SourceAccount": "920373030024"
        }
      }
    },
    {
      "Sid": "AllowCrossAccountSubscribe",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::088898720463:root" },
      "Action": ["SNS:Subscribe", "SNS:Receive"],
      "Resource": "arn:aws:sns:ap-northeast-1:920373030024:hdw-ml-alarm-topic"
    }
  ]
}
```

confused-deputy 対策の根拠:
AWS 公式 (CloudWatch `Notify_Users_Alarm_Changes` / `SNS_Confused_Deputy`) は
"most effective way to protect against the confused deputy problem is to use
the `aws:SourceArn` global condition context key with the full ARN of the
resource" としており、`aws:SourceArn` (ワイルドカード可) と `aws:SourceAccount`
の併用が推奨パターン。

## TASK-002: クライアントアカウントに HDWNotifyLogReader IAM Role を作成する

- **対応要件**: REQ-003

クライアントアカウント (920373030024) で IAM Role `HDWNotifyLogReader` を作成する。

Trust Policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::088898720463:role/hdw-notify-execution-role"
    },
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringEquals": {
        "sts:ExternalId": "<EXTERNAL_ID_HANSHIN の値>"
      }
    }
  }]
}
```

Permission Policy (inline):

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": [
      "logs:DescribeLogGroups",
      "logs:DescribeLogStreams",
      "logs:FilterLogEvents",
      "logs:GetLogEvents",
      "logs:StartQuery",
      "logs:GetQueryResults",
      "logs:StopQuery"
    ],
    "Resource": "*"
  }]
}
```

ExternalId 値の生成と取り扱い (AWS 公式 IAM User Guide 準拠):

1. 値は **自社 (HDW_Notify オペレータ)** が生成する。クライアントは生成しない。
   理由: role を引き受ける側が複数顧客間で一意性を担保する責任を持つ。
   公式記述 "It is not intended to be a 'secret'. Example Corp must provide
   the ExternalId value to each customer. What is crucial is that it must be
   generated by Example Corp and not their customers"。
2. 推奨フォーマット: 32 文字程度のランダム文字列。例:

   ```
   python -c "import secrets; print(secrets.token_urlsafe(32))"
   ```

   長さ 2-1224 / 英数字 + `+=,.@:/-` の制限内。
3. AWS 公式上は secret ではないが、攻撃面縮小のため:
   - SPEC / config YAML / git 履歴に生値を含めない
   - GHA Secret `EXTERNAL_ID_HANSHIN` に登録し、Lambda 環境変数 `EXTERNAL_ID` に
     デプロイ時投入
   - クライアントへの通達はチャットや email など監査痕跡が残る手段で実施
4. 同じ値を Trust Policy の `sts:ExternalId` 条件と Lambda の `EXTERNAL_ID` env var
   の双方に反映する。Trust Policy 編集時はマネジメントコンソールで値を直接貼る。

## TASK-003: 自社 Lambda 実行ロールに sts:AssumeRole を追加する

- **対応要件**: REQ-001, REQ-003

既存 `hdw-notify-execution-role` (088898720463) に inline policy を追加する。

Policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": "sts:AssumeRole",
    "Resource": "arn:aws:iam::920373030024:role/HDWNotifyLogReader"
  }]
}
```

他クライアント追加時は `Resource` を増やす。
現在の inline policy (`hdw-notify-permissions`) からは
`logs:StartQuery` / `GetQueryResults` / `StopQuery` の権限を削除して構わない
(これらはクライアント側 Role で実行されるため、自社ロールには不要)。
ただし Lambda 自身の実行ログ書き出し用に
`CloudWatch Logs:CreateLogGroup` / `CreateLogStream` / `PutLogEvents` は残すこと。

## TASK-004: src/main.py を SNS イベント + AssumeRole 対応に改修する

- **対応要件**: REQ-002, REQ-004, REQ-005

変更箇所:

1. Env dataclass (現 L72-105) を以下に変更:

   - 追加: `external_id: str`
   - 追加: `cross_account_role_arn: str`

2. `main()` のイベントパース (現 L233-236) を SNS event 形式に変更:

   ```python
   msg = json.loads(event["Records"][0]["Sns"]["Message"])
   alarm_name   = msg["AlarmName"]
   client_acct  = msg["AWSAccountId"]
   region       = msg.get("Region", os.environ.get("AWS_REGION", "ap-northeast-1"))
   timestamp    = msg["StateChangeTime"]
   reason       = msg.get("NewStateReason", "")
   ```

   AlarmName は SNS Message の `AlarmName` を使う (現状の
   `event["alarmArn"].split(":")[-1]` と異なる経路だが、結果は alarm 名と同じ)。

3. 時間窓導出 (現 L238-241) を REQ-005 に合わせる:
   - 既存の `cloudwatch_logs_window_before_min` / `_after_min` env var を流用
     (config YAML の値を 30 / 5 に変更)
   - `datetime.fromisoformat(timestamp.replace("Z", "+00:00"))` で TZ 保持
     (既存実装と同じ)

4. ロググループ解決:

   ```python
   log_group = env.log_group_map.get(alarm_name)
   if not log_group:
       logger.warning("no log group mapped", extra={"alarm_name": alarm_name})
       _post_minimal_embed(
           webhook_url=env.discord_webhook_url,
           alarm_name=alarm_name,
           timestamp=timestamp,
           reason=reason,
           rows_count=0,
           extra_note="LOG_GROUP_MAP に該当アラームの登録がありません",
           color=0xF1C40F,
       )
       return {"ok": True, "alarm": alarm_name, "skipped": "no_log_group_mapped"}
   ```

5. AssumeRole 呼び出しを Logs Insights 直前 (現 L246) に挿入:

   ```python
   try:
       sts_response = boto3.client("sts").assume_role(
           RoleArn=env.cross_account_role_arn,
           RoleSessionName=f"hdw-notify-{_context.aws_request_id}",
           ExternalId=env.external_id,
           DurationSeconds=900,
       )
       creds = sts_response["Credentials"]
       logger.info("assumed cross-account role",
                   extra={"assumed_role_arn": sts_response["AssumedRoleUser"]["Arn"]})
   except (BotoCoreError, ClientError) as e:
       logger.exception("assume_role failed; posting minimal fallback embed")
       _post_minimal_embed(
           webhook_url=env.discord_webhook_url,
           alarm_name=alarm_name,
           timestamp=timestamp,
           reason=reason,
           rows_count=0,
           extra_note=f"クライアントログ取得用 AssumeRole 失敗: {type(e).__name__}",
           color=0xE74C3C,
       )
       return {"ok": True, "alarm": alarm_name, "fallback": "assume_role_failed"}

   logs_client = boto3.client(
       "logs",
       region_name=region,
       aws_access_key_id=creds["AccessKeyId"],
       aws_secret_access_key=creds["SecretAccessKey"],
       aws_session_token=creds["SessionToken"],
   )
   ```

6. `start_query` 呼び出しで `logGroupName` を `log_group` に置き換える
   (`env.cloudwatch_logs_group` は削除)。

7. `_build_deeplinks_markdown` も `log_group` を引数で受け取るよう変更
   (`env.cloudwatch_logs_group` の参照を削除)。

Bedrock 呼び出し以降は変更不要 (Bedrock は自社アカウントの SDK call なので
AssumeRole 不要)。

## TASK-008: クライアントアカウントに CloudWatch Alarm を新規作成し AlarmAction を SNS Topic に指定する

- **対応要件**: REQ-002

クライアントアカウント (920373030024) で HDW_ML 監視用 CloudWatch Alarm を
新規作成する。Alarm Action として TASK-001 で作成した SNS Topic を指定する。

対象メトリクス: `HDW_Backend_Processor_0001` (HDW_ML) の `AWS/Lambda` `Errors`。
実際の threshold / evaluation period / metric statistic は `docs/operations.md`
または monitoring 担当の方針に従って確定する。本 PLAN ではアラーム配線
(Action と SNS Topic の紐付け) のみを規定する。

作成コマンド例 (threshold / period 等は要確定):

```
aws cloudwatch put-metric-alarm \
  --alarm-name hdw-backend-processor-0001-errors \
  --alarm-description "HDW_ML 失敗検知" \
  --metric-name Errors --namespace AWS/Lambda \
  --statistic Sum --period 300 --evaluation-periods 1 \
  --threshold 1 --comparison-operator GreaterThanOrEqualToThreshold \
  --dimensions Name=FunctionName,Value=HDW_Backend_Processor_0001 \
  --treat-missing-data notBreaching \
  --alarm-actions arn:aws:sns:ap-northeast-1:920373030024:hdw-ml-alarm-topic \
  --profile hanshin-t.kimura --region ap-northeast-1
```

本 SPEC は HDW_Notify (受信側) の構築仕様であり、Alarm 自体の閾値・評価窓・
運用ルールはスコープ外。Alarm のメトリクス選定が別途必要な場合は、その仕様を
独立した SPEC として起こすこと。

AlarmName は `LOG_GROUP_MAP` env var のキーと完全一致させる必要がある
(TASK-005 / `deploy/config*.yml` の `aws_log_group_map` と整合)。

## TASK-010: クロスアカウント Logs Insights 単独手動検証の手順を整備する

- **対応要件**: REQ-007

Lambda 経由の E2E (TASK-009 / REQ-006) を回す前段として、Logs Insights 経路を
単独で手動検証できる手順を整備する。本 TASK は AWS リソースの恒久変更を
伴わない (検証完了後に元状態へ戻す) ことを前提とする。

### 手順 (PowerShell 例 / 自社オペレータ端末で実施)

**1. HDWNotifyLogReader Trust Policy を一時的に拡張** (推奨ルート: AC-007-1 **(b)** — 自社オペレータ IAM User/Role を直接追加。Lambda 実行ロールへの bootstrap 前段でも検証できる)。

```powershell
# クライアントアカウント (920373030024) で実施
# 自社オペレータの呼び出し元 ARN を取得:
aws sts get-caller-identity --profile hdw-test --region ap-northeast-1
# → "arn:aws:iam::088898720463:user/<NAME>" または
#    "arn:aws:sts::088898720463:assumed-role/<ROLE>/<NAME>"

# HDWNotifyLogReader の Trust Policy を取得し、Principal.AWS に上記 ARN を
# 追加した JSON を作成。sts:ExternalId の StringEquals 条件は既存値を維持。
# Trust Policy 例の差分:
#   "Principal": {
#     "AWS": [
#       "arn:aws:iam::088898720463:role/hdw-notify-execution-role",
#       "arn:aws:iam::088898720463:user/<NAME>"          # ← 検証用に追加
#     ]
#   }
aws iam update-assume-role-policy `
  --role-name HDWNotifyLogReader `
  --policy-document file://trust-policy-with-operator.json `
  --profile hanshin-t.kimura
```

**2. ExternalId 値を環境変数に投入** (Lambda の `EXTERNAL_ID` と同値):

```powershell
# ExternalId は AWS 上 secret ではないが、ターミナル履歴に残さないため
# GHA Secret 値の手入力でセッション変数化する:
$EXTERNAL_ID = Read-Host "ExternalId" -AsSecureString | `
  ConvertFrom-SecureString -AsPlainText
```

**3. AssumeRole で一時クレデンシャルを取得**:

```powershell
$resp = aws sts assume-role `
  --role-arn "arn:aws:iam::920373030024:role/HDWNotifyLogReader" `
  --role-session-name "hdw-notify-manual-verify" `
  --external-id $EXTERNAL_ID `
  --duration-seconds 900 `
  --profile hdw-test --region ap-northeast-1 | ConvertFrom-Json
$env:AWS_ACCESS_KEY_ID     = $resp.Credentials.AccessKeyId
$env:AWS_SECRET_ACCESS_KEY = $resp.Credentials.SecretAccessKey
$env:AWS_SESSION_TOKEN     = $resp.Credentials.SessionToken
$env:AWS_REGION            = "ap-northeast-1"
```

**4. 引き受けが成立していることを確認**:

```powershell
aws sts get-caller-identity
# 期待値: Account = "920373030024"、Arn が
# arn:aws:sts::920373030024:assumed-role/HDWNotifyLogReader/hdw-notify-manual-verify
```

**5. Logs Insights クエリを発行** (`LOG_GROUP_MAP` の登録先と同じロググループに対して):

```powershell
$end   = [DateTimeOffset]::UtcNow.ToUnixTimeSeconds()
$start = $end - 3600
$qid = aws logs start-query `
  --log-group-name "/aws/lambda/HDW_Backend_Processor_0001" `
  --start-time $start --end-time $end `
  --query-string "fields @timestamp, @message | sort @timestamp desc | limit 20" `
  --query 'queryId' --output text

# ポーリング (Complete になるまで):
do {
  Start-Sleep -Seconds 2
  $r = aws logs get-query-results --query-id $qid | ConvertFrom-Json
  Write-Host "status=$($r.status)"
} while ($r.status -in @("Running","Scheduled"))

$r.status    # ← "Complete" であること (AC-007-2)
$r.results   # ← JSON 配列 (空配列でも可、rows > 0 は E2E (AC-006-3) で担保)
```

**6. 検証完了後の片付け** (恒久変更を残さない):

```powershell
# 環境変数の tmp credentials を破棄:
Remove-Item Env:AWS_ACCESS_KEY_ID, Env:AWS_SECRET_ACCESS_KEY, Env:AWS_SESSION_TOKEN
# Trust Policy から追加した自社オペレータ ARN を除去し、元の Policy に戻す:
aws iam update-assume-role-policy `
  --role-name HDWNotifyLogReader `
  --policy-document file://trust-policy-original.json `
  --profile hanshin-t.kimura
```

### 補足

本 TASK は新規コードを書かない。手順を `docs/operations.md` (または
cross-account-architecture セクションの新規サブセクション) に転記し、
構築フェーズと障害切り分けフェーズの両方から参照できるようにする。

二段 AssumeRole (AC-007-1 **(a)**) を採用する場合の差分:

- 上記 1 の代わりに、自社オペレータ IAM User の policy に
  `sts:AssumeRole → arn:aws:iam::088898720463:role/hdw-notify-execution-role` を
  一時付与し、まず execution-role を引き受け、その上で HDWNotifyLogReader を
  引き受ける。Trust Policy 改変が不要だが、自社側 IAM 権限の調整が必要。
- 通常は **(b)** の方が手数が少なく検証完了後の原状回復も容易なため、PLAN は **(b)**
  を推奨ルートとして記述する。
